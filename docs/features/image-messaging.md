## 1. 相册选择器

### 目标
集成 `expo-image-picker`，实现从相册选择图片的功能。

### 功能点
- **权限请求**：首次使用时请求相册访问权限
- **多选支持**：支持同时选择最多 5 张图片
- **格式限制**：仅支持图片格式（JPEG/PNG）
- **取消处理**：用户取消选择时优雅退出

### 技术选型
- 使用 `expo-image-picker` 库
- 支持 iOS 和 Android 双平台
- 返回 base64 编码数据用于后续上传

### 权限配置
- **Android**：`READ_MEDIA_IMAGES`（Android 13+）或 `READ_EXTERNAL_STORAGE`
- **iOS**：`NSPhotoLibraryUsageDescription` 描述声明

### UI 集成位置
在聊天输入区域的按钮组中新增相册按钮，与现有的「发送」「拍照」按钮并列显示。

## 2. 压缩 + WebSocket 上传

### 目标
压缩图片至合适大小，通过 WebSocket 发送到后端。

### 压缩策略
- **默认参数**：最大尺寸 1920×1080，质量 80%
- **智能压缩**：如果压缩后仍超过 500KB，自动降低质量和分辨率
- **格式保持**：优先 JPEG，透明通道使用 PNG
- **批量处理**：支持多张图片并行压缩

### 上传协议
通过 WebSocket 发送 `stream_data` 消息：
- `input_type` 设置为 `"image"`
- `data` 字段包含 base64 编码的图片数组
- `metadata` 包含图片尺寸、大小、格式信息

### 消息大小限制
- 单条 WebSocket 消息限制 1MB
- 超大图片自动分片发送（如需）
- 提供上传进度反馈

### 技术选型
- 使用 `expo-image-manipulator` 进行图片压缩
- 支持异步处理和取消操作

## 3. 图片识别反馈

### 目标
展示 AI 对图片的理解和处理结果。

### 依赖条件
**✅ 后端 Vision 接口已支持**

N.E.K.O 主项目已支持图片识别功能（`input_type: "camera"`），流程如下：

**前端 → 后端**：发送图片消息（`input_type: "camera"`）
**后端处理**：
1. `websocket_router.py` 接收消息
2. `core.py` 处理 `camera` 类型，调用 `stream_image()`
3. `omni_offline_client.py` 存储图片到待发送队列
4. 用户发送文本时，图片随文本一起发送给 AI（GPT-4V/Gemini）
5. AI 返回图片理解和回复

**后端 → 前端**：AI 回复消息（包含图片理解）

### UI 反馈状态
1. **上传中**：显示进度条或百分比
2. **识别中**：显示 "AI 正在识别图片..." 加载动画
3. **完成**：显示识别结果摘要（如"识别完成，共发现 3 个物体"）
4. **失败**：显示错误原因和重试按钮

### 展示位置
- 在消息气泡中显示原图缩略图
- 点击可查看大图
- AI 回复中引用相关图片

## 4. 摄像头拍照（权限 + UI）

### 目标
实现摄像头拍照功能，支持权限管理和拍照确认。

### 前置依赖
**依赖任务1（相册选择器）完成**，原因：
- 相机拍照最终也是通过相同的图片上传链路发送
- 先验证从相册选择 → 压缩 → 上传的链路正常工作
- 可排除后端问题，专注解决相机相关的技术问题

### 功能点
- **权限请求**：首次使用时请求相机权限
- **权限拒绝处理**：引导用户到系统设置开启权限
- **拍照确认**：拍摄后支持重拍或确认使用
- **数量限制**：最多同时选择/拍摄 5 张照片

### 技术选型
- 使用 `expo-image-picker` 的相机功能（无需额外依赖）
- 如需更高级的相机控制，可引入 `expo-camera`

### 权限配置
- **Android**：`CAMERA` 权限 + `hardware.camera` 特性声明
- **iOS**：`NSCameraUsageDescription` 描述声明

### UI 集成
在聊天输入区域的按钮组中新增「拍照」按钮，点击后：
1. 检查权限
2. 启动相机
3. 拍摄后添加到待发送列表
4. 用户可继续拍摄或点击发送

## 完整流程

```
用户点击相册/相机按钮
        ↓
  请求权限（首次）
        ↓
  权限被拒绝？ → 是 → 提示用户去设置
        ↓ 否
  打开选择器/相机
        ↓
  用户取消？ → 是 → 返回
        ↓ 否
  获取选中/拍摄的图片
        ↓
  压缩处理（调整尺寸、质量压缩）
        ↓
  添加到待发送列表（预览）
        ↓
  用户点击发送
        ↓
  WebSocket 上传
        ↓
  显示上传进度
        ↓
  等待后端识别结果
        ↓
  显示 AI 回复（包含图片理解）
```

---

## 错误处理策略

| 错误场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 权限被拒绝 | 引导用户到设置 | "需要权限才能访问相册/相机" |
| 图片过大 | 自动压缩 | 静默处理，显示压缩进度 |
| 压缩失败 | 尝试降低质量重试 | "正在优化图片..." |
| WebSocket 未连接 | 缓存图片，连接后重试 | "连接恢复后自动发送" |
| 上传超时 | 重试 3 次，失败后提示 | "发送失败，请重试" |
| 后端识别失败 | 显示错误信息 | "图片识别失败，请重试" |

## 性能考虑

1. **图片压缩**
   - 默认最大尺寸 1920×1080，质量 80%
   - 单张图片目标大小 < 500KB

2. **内存管理**
   - base64 数据及时清理
   - 预览使用缩略图，发送时才用完整 base64

3. **并发控制**
   - 最多同时处理 3 张图片
   - 单用户最多选择 5 张图片

4. **缓存策略**
   - 压缩后的图片临时缓存 24 小时
