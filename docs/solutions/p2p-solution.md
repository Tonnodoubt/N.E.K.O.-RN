# P2P 网络方案（局域网 + NAT 打洞）

**目标**：四层阶梯降级网络策略，最大化免费直连率，最小化服务器带宽成本。

- **综合 P2P 率（保守估算）**：约 65-75% 用户可实现零成本直连
- **综合 P2P 率（乐观估算）**：约 80-90%（含 IPv6 高普及地区）
- **兜底方案**：100% 可用，按需付费

---

## 概述

本方案通过**桌面端轻量代理 + UPnP 自动打洞**，让手机能直连电脑，无需改动主服务。

**核心原理**：
主服务必须绑定 `localhost:48911`，无法直接对外。我们在桌面端启动一个独立的 TCP 代理进程，**绑定当前 WiFi 的局域网 IP**（如 `192.168.1.10:48920`），并启用 **Token 鉴权**机制。

手机连接时按照 **四层阶梯** 逐级尝试：IPv6 直连 → 局域网直连 → UPnP TCP 打洞 → FRP 中继。每层都是零成本优先，最后才使用付费中继。

所有流量最终都通过代理转发到主服务，主服务完全感知不到变化。

**连接策略（四层阶梯降级）**

手机发起连接时，按以下优先级逐级尝试，成功则停止，失败则回退：

```
手机发起连接
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 1: IPv6 直连（零成本，延迟最低）                       │
│ 检测双方 IPv6 可用性 → 是 → IPv6 P2P直连（~30-40%实际成功率） │
└─────────────────────────────────────────────────────────────┘
    │否
    ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 2: 局域网直连（零成本，同WiFi场景）                    │
│ 检测同WiFi → 是 → 扫码直连局域网IP（10ms，100%成功）         │
└─────────────────────────────────────────────────────────────┘
    │否
    ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 3: UPnP TCP 打洞（零成本，跨网首选）                   │
│ UPnP映射公网端口 → 成功 → TCP P2P直连（30-80ms，~60%成功率） │
└─────────────────────────────────────────────────────────────┘
    │失败
    ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 3.5: STUN UDP 打洞（零成本，TCP over UDP 隧道）        │
│ UDP打洞成功 → 建立KCP隧道 → TCP流量走隧道（~70%成功率）      │
│ （可选增强，需要时实现）                                      │
└─────────────────────────────────────────────────────────────┘
    │失败
    ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 4: FRP 中继（兜底，100%成功，付费）                    │
│ 服务器转发流量（100-300ms，消耗带宽）                        │
└─────────────────────────────────────────────────────────────┘
```

---

**Layer 1: IPv6 直连**

国内移动网络和家庭宽带 IPv6 普及率已达 60-70%。如果双方都有 IPv6 地址，可直接点对点连接，无需任何打洞。

**优势**：无 NAT、零成本、延迟最低、连接最稳定。

**实际成功率修正**：

文档原始估算为 ~65%，但这是单端普及率。直连需要**双方同时具备** IPv6：
- 假设各端 IPv6 可用率 65%，双端同时可用概率 ≈ `0.65 × 0.65 ≈ 42%`
- 再扣除以下阻碍因素：
  - **操作系统防火墙**：Windows/macOS 默认阻止入站连接，用户需手动放行或程序注册防火墙规则
  - **部分运营商只分配 /128 地址**，无法在 IPv6 上监听
  - **路由器 IPv6 防火墙**：部分家用路由器默认启用 IPv6 防火墙，阻止外部主动连入
- **修正后实际成功率：约 30-40%**

**应对措施**：桌面代理启动时自动注册防火墙规则（Windows 用 `netsh`，macOS 用 `pfctl` 或应用签名自动放行）。

---

**Layer 2: 同 WiFi（局域网直连）**

用户手机和电脑连接同一个 WiFi 时，通过扫码建立连接：

```
手机（同一WiFi）        电脑
    │                    │
    │  ① 扫码二维码       │
    │  （含局域网IP:端口）  │
    │◄────────────────────┤
    │                    │
    │  ② 直接TCP连接       │
    │  192.168.1.10:48920  │
    │────────────────────►┤
    │                    │  代理转发到
    │  ③ 正常通信          │  localhost:48911
    │◄───────────────────►┤
```

**流程**：
1. 桌面端启动时，代理进程获取本机局域网 IP（如 `192.168.1.10`），绑定 `48920` 端口，同时生成随机 Token
2. 生成包含 IP、端口、Token 的二维码，用户用手机 App 扫码
3. 手机直接通过局域网 IP 连接电脑，发送 Token 验证
4. 代理验证 Token 通过后建立转发通道，延迟 <10ms，无需经过任何服务器

**成本**：零服务器成本，完全免费。

---

**Layer 3: 跨网 UPnP TCP 打洞**

当手机和电脑不在同一网络时（如手机用 4G/5G，电脑在家）：

```
手机（4G/5G）          云服务器           家用路由器           电脑
    │                    │                    │                │
    │  ① 查询公网地址      │                    │                │
    │───────────────────►│                    │                │
    │◄───────────────────┤                    │                │
    │  （无记录→直连失败）  │                    │                │
    │                    │                    │                │
    │                                         │  ② UPnP打洞    │
    │                                         │◄───────────────┤
    │                                         │  映射端口48920   │
    │                                         │                │
    │  ③ 尝试直连公网IP     │                    │                │
    │────────────────────────────────────────►┤───────────────►┤
    │  成功则 P2P 直连      │                    │                │
    │◄────────────────────────────────────────┤◄───────────────┤
```

**流程**：
1. 电脑端启动时生成随机 Token，尝试 UPnP 打洞，将内网 `48920` 映射到路由器公网端口
2. 若映射成功，上报公网地址 + Token 到云服务器
3. 手机先查询云服务器，获取公网地址和 Token
4. 手机拿到公网地址直接连接，发送 Token 验证（P2P，延迟 30-80ms）

**成本**：P2P 直连零成本。

---

**Layer 3.5: STUN UDP 打洞（可选增强）**

当 UPnP 失败时，可尝试 UDP 打洞作为备选方案：

```
手机(内网)              STUN服务器              电脑(内网)
    │                      │                      │
    │  ① 获取自己的公网地址  │                      │
    │─────────────────────►│                      │
    │◄─────────────────────┤                      │
    │  {ip:port_A}         │                      │
    │                      │◄─────────────────────┤
    │                      │  获取自己的公网地址   │
    │                      │─────────────────────►│
    │                      │                      │
    │  ② 交换地址(通过云端)  │                      │
    │◄────────────────────►│◄────────────────────►│
    │                      │                      │
    │  ③ 互相发送UDP包打洞   │                      │
    │────────────────────────────────────────────►│
    │◄────────────────────────────────────────────┤
    │  打洞成功！建立UDP会话                        │
    │                      │                      │
    │  ④ KCP隧道承载TCP流量  │                      │
    │  (TCP over UDP)      │                      │
```

**原理**：
1. 双方通过 STUN 服务器获取各自的公网 IP:端口
2. 通过云端交换地址信息
3. 互相发送 UDP 包打洞，建立会话
4. 在 UDP 上运行 KCP 协议，提供可靠传输，承载 TCP 流量

**优点**：
- 不依赖路由器 UPnP，可穿透部分 NAT
- 对 CGNAT 也有一定成功率

**缺点**：
- 需要部署 STUN 服务器
- 需要引入 KCP 等可靠 UDP 库
- 代码量增加约 1000-2000 行
- 打洞成功率约 70%（和 UPnP 接近，提升有限）

**建议**：作为可选增强层，业务上线后根据实际 UPnP 失败率决定是否实现。

---

**Layer 4: FRP 中继（兜底 + 变现）**

当前面所有 P2P 方案都失败时，自动回退到 FRP 服务器中继：

```
手机（4G/5G）          云服务器(FRP)           电脑
    │                    │                      │
    │  UPnP失败，请求中继  │                      │
    │───────────────────►│─────────────────────►│
    │                    │    服务器转发流量      │
    │◄──────────────────►│◄────────────────────►│
    │  延迟100-300ms      │                      │
```

**触发条件**：IPv6 不可用 + 不同 WiFi + UPnP 失败（CGNAT/路由器不支持等）

**成本**：消耗服务器带宽，作为增值服务收费。

**收费策略**：
- 免费用户：每月 3 小时 FRP 额度
- 付费用户：不限时 FRP 中继

**安全措施**：
- 代理只绑定局域网 IP（不是 `0.0.0.0`），减少暴露面
- **Token 鉴权**：首次连接需验证随机生成的 Token，不依赖 IP，支持手机 IP 变动
- 超时自动清理会话

**为什么简单**：
相比 WebRTC（~2000 行代码，需自定义帧协议），本方案只使用标准 TCP 连接，无需修改通信协议。相比 UDP 打洞（需要 ICE 框架和 STUN 服务器），本方案优先使用 IPv6 和 UPnP，实现简单。相比 ZeroTier/EasyTier（需安装虚拟网卡），本方案手机端零额外依赖。

**代码量估算修正**：

原始估算 ~200 行仅覆盖最简 TCP 转发。生产级实现需包含以下模块：

| 模块 | 估算代码量 | 说明 |
|------|-----------|------|
| TCP 代理 + Token 鉴权 | ~300 行 | 连接管理、Token 校验、双向转发 |
| IPv6 检测 + 防火墙适配 | ~200 行 | 多平台 IPv6 可用性检测、防火墙规则注册 |
| UPnP 映射 + 心跳续期 | ~300 行 | 端口映射、定期续期、失效检测 |
| 云端地址交换 API 客户端 | ~200 行 | 上报/查询公网地址和 Token |
| 移动端四层状态机 + 重连 | ~500 行 | 阶梯降级逻辑、超时控制、断线重连 |
| QR 码生成/解析 | ~100 行 | 二维码编解码、连接信息序列化 |
| 错误处理、日志、超时管理 | ~300 行 | 健壮性保障 |
| **合计** | **~1500-2000 行** | **不含 Layer 3.5 和 FRP 集成** |

虽然代码量比原始估算大，但仍远低于 WebRTC 方案，且全部使用标准 TCP，无需引入复杂协议栈。

## 问题

```
当前：手机 ──→ 云服务器(FRP) ──→ 电脑(localhost:48911)
           ↑                    ↑
         延迟100-300ms        主服务只能绑localhost
```

**约束**：主服务绑定 `localhost:48911` 不能改。

**思路**：启动独立代理进程，绑定 WiFi IP（如 `192.168.1.10:48920`），转发到 `localhost:48911`。

## 方案

### 架构

```
┌─────────┐      ┌──────────────────────────┐      ┌─────────────┐
│  手机    │ ──── │ 192.168.1.10:48920       │ ──── │ localhost   │
│         │      │ (LAN代理 + Token鉴权)          │      │ :48911      │
│         │      └──────────────────────────┘      │ (主服务)     │
└─────────┘              │                         └─────────────┘
      │                  │
      │    同WiFi        │    跨网
      │    ──────────────│────────────────────────────────
      │    扫码直连       │    UPnP打洞失败→FRP中继（增值服务）
      │                  │    ↓
      │              UPnP映射公网端口
      └──────────────────────────────→ 路由器公网IP:48920
```

---

### 桌面端

#### 1. 局域网代理（lan_proxy.py）

- 绑定当前 WiFi 网卡的局域网 IP（如 `192.168.1.10:48920`）
- 启动时生成随机 Token，用于首次连接验证
- 检测本机 IPv6 地址，上报到云服务器供手机查询
- 手机连接后首条消息必须携带正确 Token，验证通过才建立转发
- 将所有流量转发到 `localhost:48911`
- 由 `launcher.py` 随主服务一起启动/停止

#### 2. UPnP 打洞（upnp_mapper.py）

- 使用 UPnP 协议请求家用路由器开放公网端口
- 将内网 `48920` 映射到路由器公网 IP 的某个端口
- 获取公网 IP:端口后上报到云服务器
- 手机可从云端查询到该地址（IPv4 公网地址）

---

### 移动端

连接策略：**IPv6 → 局域网 → UPnP公网 → STUN UDP打洞 → FRP中继**，层层回退。

1. **检测 IPv6 可用性** - 免费
2. **同 WiFi 局域网直连** - 免费
3. **UPnP 公网直连** - 免费
4. **STUN UDP 打洞**（可选）- 免费
5. **FRP 中继** - 检查用户套餐，付费使用

---

### 云端地址交换服务

Layer 3（UPnP 跨网直连）和 Layer 1（IPv6 跨网直连）都依赖一个**云端地址交换服务**，用于让手机获取电脑的公网地址和 Token。这是跨网场景的关键依赖，需要单独设计。

#### API 设计

```
# 桌面端上报地址（启动时 + 地址变化时）
POST /api/v1/relay/register
{
  "device_id": "uuid-xxxx",          // 桌面端唯一标识
  "ipv6_addr": "[::1]:48920",        // IPv6 地址（如有）
  "ipv4_public": "1.2.3.4:48920",    // UPnP 映射的公网地址（如有）
  "ipv4_lan": "192.168.1.10:48920",  // 局域网地址
  "token": "random-32-byte-token",   // 本次会话的鉴权 Token
  "ttl": 60                          // 有效期（秒）
}

# 手机端查询地址
GET /api/v1/relay/lookup?device_id=uuid-xxxx
→ 返回上述地址信息（阅后即焚，查询后立即删除）

# 桌面端心跳续期
PUT /api/v1/relay/heartbeat
{
  "device_id": "uuid-xxxx"
}
```

#### 部署要求

- **存储**：Redis 内存存储，Key 为 `device_id`，TTL 60 秒自动过期
- **可用性**：服务不可用时不影响 Layer 2（同 WiFi），但 Layer 1/3 跨网场景无法工作
- **带宽**：极低，仅传输地址信息（每次请求 < 1KB）
- **安全**：HTTPS 传输，device_id 不可枚举（使用 UUID v4）

#### 备选：无服务器方案

如果初期不想维护云端服务，可用以下替代方案：
- **手动输入**：桌面端显示公网地址，用户手动复制到手机（体验差，但零成本）
- **扫码传递**：将公网地址 + Token 编码到二维码中（需要面对面扫码，不支持远程）

### 连接生命周期管理

生产环境中需要处理连接建立后的各种异常情况，这些在原始方案中未覆盖。

#### 1. 网络切换（WiFi ↔ 蜂窝）

用户从 WiFi 切换到 4G/5G（或反向）时，TCP 连接会断开。需要无感重连机制：

```
手机连接成功后
    │
    ▼
分配 session_id（32位随机字符串）
    │
    ▼
正常通信...
    │
    ▼ 网络切换，TCP 断开
    │
手机检测到断连
    │
    ▼
重新执行四层阶梯（可能从不同层成功）
    │
    ▼
连接成功后发送：{ "type": "resume", "session_id": "xxx", "token": "xxx" }
    │
    ▼
代理验证 session_id + token → 恢复会话（而非新建）
```

**关键设计**：
- `session_id` 有效期应设为 5 分钟，超时后需重新扫码
- 重连期间主服务可能有缓冲数据，代理需要处理或丢弃
- 移动端需展示 "正在重新连接..." 的 UI 状态

#### 2. NAT Keepalive

NAT 表项有超时机制（通常 TCP 为 2-5 分钟），长连接如果没有数据传输会被 NAT 设备清除：

```
保活策略：
- 每 30 秒发送一次心跳包（代理层面，对主服务透明）
- 心跳格式：固定 4 字节魔数，代理识别后不转发给主服务
- 如果 3 次心跳无响应，判定连接已死，触发重连
```

#### 3. UPnP 映射续期

路由器的 UPnP 端口映射不是永久的，可能因以下原因失效：
- 路由器重启（映射清空）
- 公网 IP 变化（运营商动态分配）
- 映射租约到期（部分路由器实现有 TTL）

```
续期策略：
- 每 5 分钟检查一次 UPnP 映射是否有效（尝试自连公网地址）
- 映射失效 → 重新映射 → 上报新地址到云端
- 公网 IP 变化 → 通知已连接的手机端进行重连
```

#### 4. 并发连接

需明确一台电脑是否支持多部手机同时连接：

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **单连接** | 新连接顶替旧连接（简单实现） | 个人用户 |
| **多连接** | 支持多部手机同时连接（需连接池管理） | 演示/教学场景 |

**建议**：V1 版本实现单连接模式，后续按需扩展多连接。

## UPnP 失效原因分析

约 30-40% 用户无法通过 UPnP 直连，主要原因如下：

### 1. CGNAT（运营商级 NAT）—— 最常见

普通家用路由器 NAT：
```
手机(192.168.1.2) → 路由器(公网IP:12345) → 互联网
```

CGNAT 双层 NAT：
```
手机(192.168.1.2) → 家用路由器(10.0.0.5) → 运营商NAT(公网IP) → 互联网
         ↑                    ↑
      你有控制权           无控制权
      （UPnP 有效）       （UPnP 无效）
```

UPnP 只能控制第一层，运营商那层 NAT 没开端口，外部无法连接。

**国内情况**：中国移动宽带大量使用 CGNAT，部分联通地区也有。

### 2. UPnP 被用户手动关闭

出于安全考虑（防止恶意软件开放端口），部分用户或企业网络会关闭 UPnP。

### 3. 路由器不支持

- 老旧/廉价路由器固件阉割了 UPnP
- 企业级路由器默认不提供 UPnP

### 4. 对称 NAT（Symmetric NAT）

某些路由器即使开了 UPnP，由于 NAT 类型为对称型，映射的端口外部无法预测。

### 覆盖率估算

| 场景 | 占比 | 结果 |
|------|------|------|
| UPnP 正常工作 | ~60% | ✅ P2P 直连 |
| CGNAT | ~20% | ❌ 只能 FRP 中继 |
| UPnP 关闭/不支持 | ~15% | ❌ 只能 FRP 中继 |
| 其他原因 | ~5% | ❌ 只能 FRP 中继 |

|------|---------|-----------|---------------|---------|

- FRP 服务器带宽需求比原始预估（5-10%）高出 3-6 倍
- 免费额度（3 小时/月）的消耗用户基数更大
- 需提前做好服务器带宽弹性扩容方案

## 安全设计

| 措施 | 说明 |
|------|------|
| **绑定特定IP** | 代理只绑定 WiFi 网卡的局域网 IP，不监听 `0.0.0.0`，其他网卡无法访问 |
| **Token 鉴权** | 桌面端生成随机 Token，手机连接后首包验证。支持 IP 变动，防止他人抢占连接 |
| **自动过期** | Token 或会话超时后自动失效，下次需重新扫码授权 |
| **无公网暴露** | 家用路由器防火墙默认阻止外部访问，只有 UPnP 主动映射的端口才对外可见 |

### 为什么不用 IP 白名单

IP 白名单有两个隐患：

1. **手机 IP 不稳定**：4G/5G 基站切换、WiFi DHCP 租期到期、MAC 地址随机化都会导致 IP 变化，用户走动一下就断连
2. **局域网内可抢占**：在校园网/公司大局域网中，他人可能在你扫码前抢先连接，导致你无法连上

**Token 鉴权流程**：
```
桌面端启动
    │
    ▼
生成随机 Token（如 32 位随机字符串）
    │
    ├──► 局域网场景：Token 嵌入二维码
    └──► 跨网场景：Token 上传云端，手机查询时下发
    │
手机连接代理后，首条消息发送 Token
    │
代理验证 Token 正确 → 建立转发通道
代理验证失败 → 立即断开连接
```

无论手机 IP 怎么变，只要有正确 Token 就能连接，且绝对安全。

## 安全注意事项

本方案已实现 **Token 鉴权** 和 **IP 绑定**，但仍需注意以下安全事项：

### 1. 传输加密确认

Token 仅用于身份验证，不加密传输内容：

| 主服务协议 | 风险等级 | 建议 |
|-----------|---------|------|
| HTTPS/WSS | 低 | 代理仅转发加密流量，无需额外处理 |
| HTTP/WS | 中 | 建议主服务升级 TLS，或在代理层增加 TLS 终止 |

**注意**：如主服务为明文传输，公网 WiFi 或运营商网络存在中间人风险。

### 2. 限流与防 DoS

代理层建议实现以下防护：

```
失败尝试限制：
- 同一 IP 连续 3 次 Token 错误 → 加入黑名单 10 分钟
- 同一 IP 每秒超过 10 个连接 → 临时限流

资源保护：
- 最大并发连接数限制（如 100）
- 单连接最大空闲时间（如 5 分钟）
```

**注意**：Token 本身为 256 位随机值，暴力破解不可行，但需防范连接池耗尽攻击。

### 3. 云端数据最小化

跨网打洞时，公网地址和 Token 需临时存储：

```
推荐做法（阅后即焚）：
- 使用 Redis 内存存储，不落地硬盘
- TTL 设置 60 秒，过期自动删除
- 手机查询成功后，立即删除云端记录
- 传输使用 HTTPS，防止中间人截获
```

**风险**：若云端数据库被攻破，攻击者可获得有效 Token 和 IP，直接连接用户电脑。

### 4. IPv6 与 UPnP 暴露面

| 场景 | 暴露范围 | 防护 |
|------|---------|------|
| IPv6 直连 | 全球可达 | Token 鉴权是唯一防护，需确保 Token 强度 |
| UPnP 映射 | 公网端口开放 | 路由器防火墙 + Token 双重保护 |
| 局域网直连 | 仅同 WiFi | 相对安全，但仍需 Token 防止内网抢占 |

**建议**：
- 代理进程不以 root 权限运行
- 定期（如每次启动）更换 Token
- 生产环境部署前进行渗透测试
- **IPv6 场景下代理端口随机化**：不使用固定端口 48920，而是每次启动随机选择高位端口（49152-65535），降低被扫描命中的概率
- **建议在代理层增加可选 TLS**：即使主服务是明文，代理层可做 TLS 终止，防止公网传输中 Token 和数据被中间人截获。IPv6 和 UPnP 场景下流量经过公网，这一点尤为重要

## 限制

| 层级 | 场景 | 结果 | 占比（保守） | 占比（乐观） |
|------|------|------|-------------|-------------|
| Layer 1 | 双方都有 IPv6 且防火墙放行 | ✅ 直连(<10ms) | ~35% | ~45% |
| Layer 2 | 同WiFi | ✅ 直连(10ms) | ~10%（场景有限）| ~10% |
| Layer 3 | UPnP正常工作（非CGNAT、非对称NAT）| ✅ 直连(30-80ms) | 剩余中~50% ≈ +27% | 剩余中~60% ≈ +27% |
| Layer 3.5 | STUN UDP打洞成功 | ✅ 直连(30-80ms) | ~5%（可选增强）| ~10% |
| Layer 4 | 所有P2P失败 | ⚠️ 回退FRP(100-300ms) | ~25-30% | ~15-20% |

**综合 P2P 率**：
- 基础四层（不含 Layer 3.5）：保守约 **65-75%**，乐观约 **80%** 免费直连
- 含 UDP 打洞增强：保守约 **70-80%**，乐观约 **85-90%** 免费直连

**注意**：以上为全国平均水平。地域差异显著：
- 一线城市 + 新小区（IPv6 + 独立公网 IP）：P2P 率可达 90%+
- 三四线城市 + 移动宽带（CGNAT 高发区）：P2P 率可能仅 50-60%

剩余约 15-30% 用户需使用付费 FRP 中继。

所有情况都能连上，免费用户有额度限制，付费用户无限制。

## 推荐实施路径

建议按阶段递进实现，每阶段独立可交付：

### Phase 1：同 WiFi 直连（MVP，1-2 周）

**目标**：验证代理架构，覆盖最核心的使用场景。

**交付物**：
- TCP 代理进程（绑定 LAN IP:48920，转发到 localhost:48911）
- Token 鉴权（生成 + 验证）
- 二维码生成（包含 LAN IP + 端口 + Token）
- 移动端扫码连接

**验证标准**：手机与电脑在同一 WiFi 下，扫码后 10 秒内建立稳定连接。

### Phase 2：IPv6 直连 + UPnP 打洞（2-3 周）

**目标**：覆盖跨网场景，实现 65-75% P2P 率。

**交付物**：
- IPv6 可用性检测 + 防火墙规则注册
- UPnP 端口映射 + 续期
- 云端地址交换服务（最小化版本）
- 移动端四层状态机

**验证标准**：手机用 4G 连接家中电脑，Layer 1 或 Layer 3 至少一个成功直连。

### Phase 3：FRP 兜底 + 商业化（2-3 周）

**目标**：100% 连接成功率，商业模式落地。

**交付物**：
- FRP 中继集成
- 用户套餐系统（免费额度 + 付费）
- 连接方式检测与 UI 展示
- 网络切换重连机制

### Phase 4：可选增强（按需）

- Layer 3.5 STUN/KCP UDP 打洞
- 代理层 TLS 加密
- 多设备并发连接
- 连接质量监控与数据埋点

**总周期预估**：Phase 1-3 合计约 5-8 周，即可交付一个功能完整的 P2P 网络方案。
